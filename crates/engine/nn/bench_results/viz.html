nn — performance dashboard

Per-shape benchmarks of the in-house neural-net library and several peer Rust libraries. Generated by cargo run -p nn-bench --example <driver> + cargo run -p nn --example build_viz. Schema v2; see docs/designs/nn_benchmark_harness.md for the design.
how to read this dashboard ▾

what each label means

labelwhat it iskernel notes
nn-scalar this repo's crates/ml/nn, default CPU build. scalar Rust loops; LLVM autovectorises on AVX2. Single-threaded.
nn-simd same library, --features simd. portable f32x8 via the wide crate. Single-threaded.
nn-parallel same library, --features parallel. multi-threaded sgemm via the gemm crate (Faer). Same crate candle uses internally. NOT bit-deterministic — gated; determinism tests don't run with this feature on.
nn-gpu-oneshot same library, --features gpu, slice-based matmul() API. cubecl/wgpu naive kernel. Every call uploads inputs + allocates output + dispatches + syncs + downloads. ~1 ms/call regardless of shape because of the round-trip overhead.
nn-gpu-pooled same library, GpuContext API. Uploads run ONCE outside the timing loop; output buffers come from an in-process pool (99% reuse in practice). Same kernel as oneshot, no host↔device traffic in the loop. 2-8× faster than oneshot depending on shape; matches an honest "what does the kernel itself cost?" measurement.
ndarray-mm baseline: ndarray + matrixmultiply::sgemm. hand-tuned single-threaded sgemm (bluss). NOT a neural-net lib — no autograd. Isolates "how much of the gap is just the matmul kernel?"
candle candle-core from Hugging Face, CPU. uses the gemm crate — multi-threaded sgemm. Architecturally closest to nn.
candle-cuda same candle, --features candle-cuda. matmul dispatches to cuBLAS (NVIDIA's hand-tuned f32 sgemm); elementwise to candle's PTX kernels. Sets the realistic GPU upper bound. Synchronised between calls so the numbers reflect actual compute, not async queue-insert.
burn burn-ndarray backend from tracel-ai. same multi-backend framework as cubecl; here on the ndarray CPU backend.
dfdx dfdx by coreylowman. compile-time shape checking via const generics. Tests whether tape overhead matters at small shapes.
tch tch-rs — Rust bindings to libtorch (PyTorch C++). upper bound for "what's actually fast". Multi-threaded MKL/oneDNN paths.

metrics

metricmeaning
mean µswall-clock per call; lower is better.
GFLOP/sbillion floating-point ops per second; for matmul = 2·m·k·n / mean_time. Higher is better. Theoretical peak on AVX2-FMA: ~64 GFLOP/s single-core single-precision.
itersnumber of calls measured. Time-budgeted: fast ops get tens of thousands, big ops a handful.
CPU% max / RAM% max / GPU% maxwhole-machine resource max during the bench (sampled every 50 ms via sysinfo + NVML). For single-threaded code on N cores, the CPU floor is ~100/N%; numbers significantly above mean another process was busy.

machine

headline

matmul throughput vs problem size

GFLOP/s per shape, log-x axis on problem size m·k·n. Higher is better. Each line is one bench configuration (one JSON file in bench_results/).

SIMD speedup over scalar

Ratio scalar mean / simd mean per matmul shape. >1 means SIMD won; <1 means scalar (LLVM autovec) won.

train_step latency vs work

Microseconds per full forward + backward + Adam step, vs batch · hidden. A single click in the textbook widget triggers one of these.

matmul mean latency

Microseconds per matmul; lower is better. Symmetric to the GFLOP/s curve but useful when the matrix has unusual aspect ratio.

elementwise ops — per-call cost vs vector size

Each chart times one operation called once on a vector of n floats. Both axes are log scale, so a 45° slope = linear scaling (twice the elements ≈ twice the time). A flat-then-rising shape means a fixed per-call overhead dominates at small n. Vertical spread between libraries at a given n reads as a ratio, not a difference. Lower is better.

add — pairwise sum, c[i] = a[i] + b[i]

Simplest binary op; the cost floor.

mul — pairwise product, c[i] = a[i] · b[i]

Same shape as add; small differences here come from FMA scheduling.

relu — max(x, 0) per element

Compare-and-zero; one branch, no transcendental. Usually the cheapest activation.

sigmoid — 1 / (1 + exp(−x)) per element

Needs exp — the only chart here that exposes the lib's transcendental quality. Expect ~5–10× of relu.

system resource pressure during each bench

These metrics are whole-machine, not per-process — background apps, OS daemons, browser tabs, all included. High outliers here mean the timing data for that row may be noisy (something else was using the machine). Cross-reference with the charts above when interpreting a result.

CPU utilization (max during bench)

Whole-system CPU% averaged across all logical cores. For single-threaded code on an N-core machine, the floor is ~100/N%; numbers significantly above that mean another process is busy.

system memory used (% of total RAM)

Total RAM in use as a percent of installed RAM. A stable value across rows means our benches don't allocate much; a rising trend means we're leaking or stressing the allocator.

GPU utilization (NVIDIA only)

Whole-system GPU% from NVML. Currently the nn library runs CPU-only, so any non-zero value here is from another process — useful "is something else on the box?" signal. When the cubecl GPU backend lands, this is where the payoff will show up.

system thread count

Total threads alive on the machine, summed across every process. The nn benchmarks are single-threaded; spikes here mean someone else spawned threads.
